# 职通车开发与代码审查作战手册

本手册基于已安装的 OpenAI 官方技能整理，目的是把“能跑”提升到“可交付、可维护、可复审”。

## 已安装技能

- `doc`
- `pdf`
- `playwright`
- `screenshot`
- `security-best-practices`
- `security-threat-model`

> 说明：Codex 需要重启后才能把新技能作为会话内可触发技能正式加载。

## 适用于职通车的推荐工作流

### 1. 功能开发前

- 先明确变更范围：是后端接口、前端交互、AI 链路，还是文档与交付物。
- 涉及上传、鉴权、云同步、模型调用时，优先按 `security-best-practices` 思路先做安全设计再编码。
- 涉及跨组件改动时，先按 `security-threat-model` 思路写出资产、入口、边界和攻击面。

### 2. 后端变更

- 新增或修改 FastAPI 接口时，检查：
  - 是否需要鉴权
  - 是否需要限流
  - 是否需要大小/类型校验
  - 是否会触发高成本外部请求
  - 是否会泄露敏感日志
- 对外部抓取类逻辑，优先保证：
  - 独立超时
  - 降级可见
  - 错误不缓存
  - 证据链可追溯

### 3. 前端交互与视觉质量

- 每次改完 UI，至少做一轮：
  - `npm run build`
  - 本地页面打开
  - 关键流程点击冒烟
- 重度依赖 `playwright` 与 `screenshot`：
  - `playwright` 用于表单、弹窗、上传、分页、错误态冒烟
  - `screenshot` 用于视觉回归、文案溢出、表格布局、暗色背景可读性检查
- 视觉交付标准：
  - 文字不溢出
  - 空态/错误态完整
  - 按钮文案明确
  - loading / degraded / success 三态都能自洽
  - 表格、卡片、弹窗在 1440px 和 390px 下都可用

### 4. PDF / 简历链路

- 触碰简历上传、解析、导出、预览时：
  - 用 `pdf` 技能检查布局和可读性
  - 如有对外交付文档，再用 `doc` 技能检查版式一致性
- 禁止只靠文本抽取判断交付质量，必须同时检查渲染结果。

### 5. 代码审查清单

每次提交前，至少过以下清单：

- **构建**
  - 前端 `npm run build` 通过
  - 后端关键接口 smoke test 通过
- **安全**
  - 无新增 `* + credentials` CORS
  - 无明文 secret / token
  - 无未鉴权的高风险接口
  - 无未限制的文件解析入口
- **稳定性**
  - 外部 API 有超时
  - 降级路径明确
  - 缓存不固化错误结果
- **产品**
  - 用户可理解错误信息
  - 空态不突兀
  - 交互路径不依赖隐藏前提
  - 文案无乱码

## 当前项目建议的固定命令

- 前端构建：`cd temp_frontend && npm run build`
- 后端冒烟：`python - <<'PY' ... PY` 或使用 `FastAPI TestClient`
- AI 雷达回归：直接调用 `/api/analyze/ai-radar`
- 真实浏览器冒烟：`npx --yes --package @playwright/cli playwright-cli --help`

## 当前基线结论

- AI 人才雷达主链路已恢复可用，公开样本实测可返回正常分数与证据。
- 前端当前可构建。
- 项目仍需继续推进会话安全、限流、远程文件护栏三项工程化治理，达到更稳妥的生产可交付水平。
